# Books-to-Skill 元方法

把经典书籍中的方法论提炼为 agent 可直接使用的结构化 skill。这是一个"造武器的武器"。

---

## 核心资产

### 核心书库

预先整理的精选书单，覆盖 16 个领域、90 本经典书籍：

```
assets/core_library.json
```

每本书包含：id、书名（中英）、作者、出版年份、核心框架名称、方法论标签、最佳适用场景。

**16 个领域**：营销与品牌、战略咨询、管理与组织、销售与谈判、产品与创新、领导力、个人成长与效能、沟通与表达、思维与决策、创业、人力资源与人才、运营与供应链、设计与用户体验、数据与增长、应用心理学、财务与投资。

### 已完成的范例

营销与品牌领域已经完成了完整的 skill 打样，可作为标准参考：

```
../marketing/SKILL.md           → 索引 skill（场景识别、方法论匹配、三种使用模式）
../marketing/references/*.md    → 9 个方法论 reference 文件
```

## 生产流程

### 第 0 步：选择领域

从核心书库中选择一个领域。加载 `assets/core_library.json`，查看该领域的书单。

如果用户想添加核心书库中没有的书或领域：
1. 使用书籍 search API 验证书籍真实性（书名、作者、出版信息）
2. 判断这本书是否有"可提炼的操作框架"——有些书有深刻洞见但没有可操作的框架，不适合做 skill
3. 如果合适，添加到书库并继续

### 第 1 步：为每本书提炼方法论（生成 reference 文件）

这是最核心的步骤。每本书提炼为一个 reference 文件，遵循统一结构。

#### reference 文件标准模板

```markdown
# [方法论中文名] ([方法论英文名])
> 来源：《[书名]》— [作者]（[出版年份]）
> 延伸：《[相关书籍]》（如有）

## 一句话定位
这个方法论解决什么问题？一句话说清楚它是什么、对谁有用。
要求：具体、不空泛、有场景感。

## 核心框架
理论的核心模型，结构化呈现。
- 如果是步骤型（如定位四步法）：按步骤展开
- 如果是维度型（如 STEPPS）：按维度展开
- 如果是矩阵型（如 Radical Candor）：用表格/矩阵展开
每个步骤/维度要有：定义 + 关键问题 + 实操手段

## 怎么用（操作指南）
按用户可能遇到的 2-3 个真实场景拆分，每个场景写成：
### 场景 X：[场景描述]
输入：[用户带来的是什么]
执行步骤：[agent 一步步怎么做]

关键原则：
- 场景是从用户任务出发的，不是从理论章节出发的
- 步骤必须可执行，不是"你可以考虑……"
- 每个场景都要有明确的输入和输出

## 判断标准
用 checklist 格式列出"做得好的标准是什么"。
让 agent 和用户都能用来检查输出质量。

## 常见误区
列出 3-5 个这个理论最容易被误用的地方。
这是专家和新手的核心差距——专家知道什么时候不该用。

## 组合使用
这个方法论和同体系内其他方法论的配合关系。
格式：[本理论] + [另一理论]：[配合方式和场景]

## Agent 指令
给 agent 的执行指南，包含：
### 信息收集
在执行前需要知道什么（一般 3-5 个关键信息）
### 输出格式
结构化的输出模板，让输出规范且可预期
### 注意事项
执行中的红线、边界条件、语气要求
```

#### 提炼质量标准

一个好的 reference 文件必须满足：

1. **可操作**：读完"怎么用"部分，agent 能直接执行，不需要回去翻原书
2. **有场景**：不是理论概述，是"当用户说XXX的时候，agent做YYY"
3. **有判断**：不只是说"怎么做"，还要说"怎么判断做得好不好"和"什么时候不该用"
4. **有连接**："组合使用"部分让这个方法论不是孤立的，而是方法论网络的一个节点
5. **有人味**：用大白话写，不堆术语。用户读起来像在跟一个懂行的朋友聊天
6. **中英并存**：方法论名称保留英文原名 + 中文翻译，正文根据用户语言自然切换

#### 提炼中的常见错误

- ❌ 写成读书笔记/书评（"这本书讲了……作者认为……"）
- ❌ 照搬书中案例而不提炼出可迁移的框架
- ❌ 只有"是什么"没有"怎么用"
- ❌ "怎么用"太抽象（"根据情况灵活运用"）
- ❌ 忽略"什么时候不该用"

### 第 2 步：生成索引 SKILL.md

每个领域一个索引 skill，负责：

#### 索引 SKILL.md 标准结构

```markdown
---
name: [domain]-playbook
description: >
  [领域]方法论智库。[触发条件描述，包含场景关键词和理论名称关键词]
---

# [领域]方法论智库

## 方法论清单
| 编号 | 方法论 | 来源 | 一句话定位 | 最佳场景 |
表格形式列出所有方法论。

## 使用方式

### 模式一：用户有具体任务（实战模式）
识别任务 → 匹配方法论 → 加载 reference → 执行 → 输出结果

### 模式二：用户让 Agent 学习/进修（装备模式）
加载全部 reference → 内化 → 汇报学习成果（每个理论一句话说明能用它做什么）

### 模式三：用户指定理论
直接加载对应 reference → 执行

## 场景 → 方法论映射表
| 用户场景 | 首选方法论 | 补充方法论 |
这是索引 skill 最核心的价值——让 agent 不需要"思考"就能匹配到对的理论。

## 加载 Reference 文件
列出所有 reference 文件路径。强调按需加载，不要全部一次性加载。

## 行为准则
1. 框架是手段，用户的任务是目的
2. 不要学术化，用大白话
3. 输出要可执行
4. 中英双语自然切换
5. 鼓励组合使用
6. 承认局限
```

#### description 的写法要点

索引 SKILL.md 的 description 是触发的关键。要做到：
- 列出该领域所有常见的用户场景关键词
- 列出该领域所有经典理论名称（中英文）
- 说明"即使用户没有提到XX这个词，只要任务本质是XX相关，也应该触发"
- 比正常的 description 更"推"——宁可过度触发也不要漏触发

### 第 3 步：场景映射表设计

这是每个领域 skill 最需要人工判断的部分。

设计原则：
1. **场景来自用户的真实表述**，不是理论术语。写"怎么让我的产品卖得更好"而不是"转化率优化"
2. **每个场景有首选+补充**，首选是主力，补充是配合
3. **覆盖要全**——把该领域用户可能说的话都想到
4. **不重叠**——每种用户表述只有一个最佳匹配

### 第 4 步：测试与验证

用 3-5 个典型 prompt 测试：

1. **装备模式测试**："你去学一下[领域]方法论"
   - 检查：是否正确加载所有 reference？汇报是否体现"真的理解了"？
   
2. **实战模式测试**：给一个具体任务
   - 检查：是否正确匹配了方法论？执行是否按 reference 的 Agent 指令走？输出是否结构化、可执行？
   
3. **指定理论测试**："用[理论名]帮我做XXX"
   - 检查：是否直接加载了正确的 reference？

4. **边界测试**：给一个该领域边缘/不适用的任务
   - 检查：是否诚实说明局限，而不是强行套用？

## 批量生产工作流

当需要为多个领域批量生产 skill 时：

### 优先级排序

建议按以下顺序生产（根据方法论密度和用户需求）：

| 优先级 | 领域 | 理由 |
|--------|------|------|
| ✅ 已完成 | 营销与品牌 | 打样完成 |
| P0 | 战略咨询 | 方法论密度最高，框架即产品 |
| P0 | 销售与谈判 | 直接帮用户赚钱，获得感强 |
| P1 | 产品与创新 | 产品经理群体需求大 |
| P1 | 管理与组织 | 企业客户核心需求 |
| P1 | 领导力 | 高管教练场景 |
| P2 | 沟通与表达 | 通用能力，跨领域适用 |
| P2 | 思维与决策 | 通用能力，跨领域适用 |
| P2 | 创业 | 和产品/战略有交叉 |
| P3 | 个人成长与效能 | 偏个人，非商业核心 |
| P3 | 人力资源 | 垂直领域 |
| P3 | 运营与供应链 | 垂直领域 |
| P3 | 设计与UX | 垂直领域 |
| P3 | 数据与增长 | 垂直领域 |
| P3 | 应用心理学 | 偏基础理论 |
| P3 | 财务与投资 | 需要特别注意免责 |

### 单领域生产时间预估

- 选书确认：5 分钟
- 每个 reference 文件：10-15 分钟
- 索引 SKILL.md：10 分钟
- 测试验证：10 分钟
- 总计：一个领域（5-6 本书）≈ 80-100 分钟

## 用户自助扩展

如果用户想添加核心书库之外的书：

1. 用户提供书名和作者
2. 通过书籍 search API 验证书籍信息
3. 判断是否适合提炼（有操作框架 vs 纯洞见）
4. 如果适合，按标准模板生成 reference 文件
5. 更新对应领域的索引 SKILL.md

如果用户想添加核心书库之外的领域：

1. 明确这个领域的范围和典型场景
2. 选 5-8 本经典书籍（优先选有明确框架的）
3. 按标准流程生产全套 skill
4. 更新核心书库
