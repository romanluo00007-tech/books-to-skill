# 精益创业 (The Lean Startup)
> 来源：《The Lean Startup》— Eric Ries（2011）
> 延伸：《Running Lean》— Ash Maurya（2012）

## 一句话定位

别闷头做一年再上线——用最小可行产品快速验证假设，用数据而非直觉决定下一步。适合新产品验证、创业早期、大公司内部创新项目。

## 核心框架

### 构建-测量-学习循环 (Build-Measure-Learn Loop)

这是精益创业的核心引擎：

| 阶段 | 做什么 | 关键产出 |
|------|--------|---------|
| **构建 (Build)** | 把假设变成最小可行产品（MVP） | 一个能测试核心假设的产品/原型 |
| **测量 (Measure)** | 用数据看 MVP 的表现 | 可行动指标（非虚荣指标） |
| **学习 (Learn)** | 基于数据决定：坚持还是转型？ | 经过验证的认知（Validated Learning） |

**关键**：这个循环要尽可能快地转。一个循环从几周到几个月，绝对不是几年。速度就是生命。

### 最小可行产品 (MVP)

MVP 不是"功能最少的产品"，而是"能验证核心假设的最小实验"。

| MVP 类型 | 做什么 | 适用场景 |
|---------|--------|---------|
| **视频 MVP** | 拍一个产品演示视频看反应 | Dropbox 最早就是一个演示视频 |
| **手动 MVP (Wizard of Oz)** | 前台看起来自动化，后台全靠人工 | 验证需求是否存在 |
| **着陆页 MVP** | 做一个产品页面，看多少人点"注册" | 验证市场兴趣 |
| **单功能 MVP** | 只做最核心的一个功能 | 验证核心价值 |
| **众筹 MVP** | 在众筹平台测试愿意付费的人有多少 | 验证付费意愿 |

### 可行动指标 vs 虚荣指标

| 虚荣指标（看起来好看但没用） | 可行动指标（能指导决策） |
|---------------------------|----------------------|
| 总注册用户数 | 每周活跃用户数 |
| 页面浏览量 | 转化率 |
| 总收入 | 单位经济模型（LTV/CAC） |
| 下载量 | 留存率 |

### 转型 (Pivot)

当数据证明当前方向不对时，系统性地改变策略：

| 转型类型 | 改什么 |
|---------|--------|
| **客户群转型** | 换一群目标客户 |
| **需求转型** | 解决不同的问题 |
| **渠道转型** | 换一种获客方式 |
| **技术转型** | 用不同技术实现同样的价值 |
| **放大/缩小转型** | 把一个功能变成整个产品，或反过来 |
| **商业模式转型** | 换盈利方式 |

**关键原则**：转型不是失败，是"用数据发现了更好的方向"。

## 怎么用（操作指南）

### 场景 A：验证一个新产品/功能想法

输入：用户有一个产品创意，想知道是否值得做

执行步骤：
1. 明确核心假设——这个创意成立的前提是什么？（用户有这个需求？愿意付费？能找到用户？）
2. 排列假设的风险——哪个假设如果错了，整个创意就不成立？先验证它
3. 设计 MVP——能验证最高风险假设的最小实验是什么？
4. 定义成功标准——什么数据说明假设成立？什么说明不成立？**提前定好**
5. 规划测试——需要多长时间？接触多少用户？
6. 输出验证计划

### 场景 B：产品数据不好，判断是坚持还是转型

输入：用户的产品已上线但数据不理想

执行步骤：
1. 区分虚荣指标和可行动指标——先确认看的是对的数据
2. 检查核心假设——最初的假设是什么？数据在说哪个假设有问题？
3. 判断是优化还是转型——如果核心价值假设错了，优化没有意义
4. 如果需要转型，分析哪种转型最有可能——客户群？需求？渠道？
5. 设计下一轮 MVP 来测试新假设
6. 输出决策建议（坚持+优化方向 或 转型+新假设）

### 场景 C：为创业项目设计精益验证路线图

输入：用户在创业早期，想做整体的验证规划

执行步骤：
1. 列出所有核心假设（需求假设、增长假设、盈利假设）
2. 按风险排序——最可能错的先验证
3. 为每个假设设计对应的 MVP 和成功标准
4. 规划 3-4 个验证循环，每个循环 2-4 周
5. 设定"转型还是坚持"的决策点
6. 输出精益验证路线图

## 判断标准

- [ ] **假设明确** — 清楚列出了"这个创意成立需要什么条件为真"
- [ ] **MVP 足够小** — 不是做了半个产品，是做了一个实验
- [ ] **成功标准提前定** — 不是看了数据再定义"好不好"，而是事先说好
- [ ] **用可行动指标** — 不被虚荣指标误导
- [ ] **有转型意识** — 数据不好时敢于改方向，不死守沉没成本

---

## 常见误区

1. **MVP 做成了"缩水版产品"**。MVP 不是砍功能砍到最少，是"验证假设的最小实验"。一个着陆页也可以是 MVP。
2. **没有定义成功标准就开始测**。先跑数据再定义"好不好"，你的大脑会替你合理化任何结果。标准必须提前定。
3. **只看虚荣指标就觉得方向对了**。"下载量涨了"≠ "产品有价值"。看留存、看付费转化、看复购。
4. **不敢转型**。"已经做了6个月了，不能白费"——这是沉没成本谬误。数据告诉你方向错了，越早转越好。
5. **无限循环不决策**。精益的循环是为了快速到达决策点，不是无限"再测一轮"。每个循环结束要有明确的"坚持还是转型"判断。

## 组合使用

- **精益创业 + 妈妈测试**：精益的循环需要用户数据，妈妈测试教你怎么正确地从用户那里获取真实信息（而不是礼貌的谎言）。
- **精益创业 + 设计冲刺**：Sprint 可以作为构建-测量-学习循环中"构建"阶段的加速器——5天内完成从创意到原型到测试。
- **精益创业 + 跨越鸿沟**：精益帮你找到产品-市场契合，跨越鸿沟帮你从早期市场扩展到主流市场。先精益验证，再考虑跨越鸿沟。
- **精益创业 + JTBD**：JTBD 帮你找到"用户真正要完成的任务"，精益帮你验证"你的方案是否能帮他完成"。

## Agent 指令

### 信息收集
在执行前需要知道：
- 用户的产品/创意是什么？
- 目前在什么阶段——纯想法、有原型、已上线？
- 最大的不确定性是什么？（不确定有没有需求？不确定能不能获客？不确定用户愿不愿意付费？）
- 有没有已有的数据？
- 时间和资源约束——有多少时间/预算来做验证？

先问阶段和最大不确定性，其他在过程中确认。

### 输出格式

**验证计划**输出：
```
🧪 [产品名] 精益验证计划

📋 核心假设（按风险排序）：
1. 🔴 [最高风险假设] — 如果错了，整个方向不成立
2. 🟡 [中等风险假设]
3. 🟢 [较低风险假设]

🛠️ MVP 设计：
- 验证假设：[#1]
- MVP 类型：[着陆页/手动/单功能/……]
- 做什么：[具体描述]
- 需要多久：[时间]

📊 成功标准（提前定义）：
- ✅ 假设成立：[什么数据说明成立]
- ❌ 假设不成立：[什么数据说明不成立]
- 🔄 灰色地带：[如果数据在中间，怎么办]

🗓️ 时间线：
- Week 1：[……]
- Week 2：[……]
- Week 3：[……]
- 决策点：[坚持/转型/需要更多数据]
```

### 注意事项
- "假设"要帮用户从模糊变清晰——很多人说不清自己的核心假设是什么
- MVP 要真的最小——用户常常想做太多，要帮他们砍到最核心
- 成功标准必须提前定——这是精益最容易被跳过、也最重要的一步
- 不要让用户陷入"分析瘫痪"——精益的精神是"动手比想更重要"
