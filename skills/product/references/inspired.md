# 启示录 (Inspired)
> 来源：《Inspired: How to Create Tech Products Customers Love》— Marty Cagan（2008/2018第二版）

## 一句话定位

产品管理的"圣经"——教你如何组建产品团队、做好产品发现、区分"做什么"和"怎么做"。适合产品经理、产品总监、任何需要系统化做产品的人。

## 核心框架

### 产品发现 vs 产品交付

| 维度 | 产品发现 (Discovery) | 产品交付 (Delivery) |
|------|---------------------|-------------------|
| 核心问题 | 做什么？值不值得做？ | 怎么做？做出来 |
| 目标 | 找到既有价值又可行的产品方案 | 高质量地实现方案 |
| 风险 | 价值风险、可用性风险、可行性风险、商业可行性风险 | 交付风险 |
| 方法 | 原型、用户测试、实验 | 敏捷开发、Sprint |
| 谁主导 | 产品经理 + 设计师 + 技术领导 | 工程团队 |

**关键洞察**：大部分团队把 90% 的时间花在交付上，只有 10% 在发现上。但真正的浪费不是"做得慢"，而是"做了不该做的东西"。

### 四种风险

每个产品决策都要评估四种风险：

| 风险 | 核心问题 | 谁负责 |
|------|---------|--------|
| **价值风险 (Value)** | 用户会用吗？解决了真实需求吗？ | 产品经理 |
| **可用性风险 (Usability)** | 用户会用吗？能学会吗？ | 设计师 |
| **可行性风险 (Feasibility)** | 我们做得出来吗？技术上可行吗？ | 工程师 |
| **商业可行性风险 (Business Viability)** | 对公司有利吗？法务/财务/销售都支持吗？ | 产品经理 + 利益相关者 |

### 赋能型产品团队 (Empowered Product Team)

| 角色 | 职责 | 关键能力 |
|------|------|---------|
| **产品经理** | 决定做什么、为什么做 | 深度理解用户、理解商业、理解数据 |
| **产品设计师** | 定义用户体验 | 用户研究、交互设计、原型 |
| **技术领导** | 判断可行性、提出技术方案 | 技术深度、系统思维 |

**关键原则**：
- 三个角色是"传教士团队"（Missionaries），不是"雇佣兵团队"（Mercenaries）
- 团队被赋予要解决的问题，而不是被交待要做的功能
- 团队拥有解决方案的自主权

### 产品发现技术

| 技术 | 用途 | 适用阶段 |
|------|------|---------|
| **用户访谈** | 理解用户需求和痛点 | 早期探索 |
| **客户拜访 (Customer Discovery)** | 在真实环境中观察用户 | 需求验证 |
| **原型测试** | 用低保真/高保真原型测试方案 | 方案验证 |
| **A/B 测试** | 用数据比较不同方案 | 优化阶段 |
| **可用性测试** | 看用户能否顺利完成任务 | 方案细化 |
| **参考客户计划** | 找 6 个种子客户深度合作验证 | 新产品验证 |

## 怎么用（操作指南）

### 场景 A：做产品发现——决定下一步做什么

输入：用户是产品经理，需要决定做什么功能/产品

执行步骤：
1. 明确要解决的问题（不是"要做的功能"）
2. 用四种风险框架评估：价值有风险吗？可用性有风险吗？技术可行吗？商业上行吗？
3. 针对风险最高的维度设计验证方式——价值风险高就做用户访谈，可用性风险高就做原型测试
4. 设计产品发现计划——用什么方法、多长时间、和谁一起
5. 定义"什么证据能让我们有信心开发"
6. 输出产品发现计划

### 场景 B：评估/优化产品团队的工作方式

输入：用户是产品负责人，觉得团队效率低或方向不对

执行步骤：
1. 诊断团队现状——是"传教士团队"还是"雇佣兵团队"？
2. 检查发现/交付比例——花了多少时间在发现上？如果不到 30%，这是核心问题
3. 检查三个角色是否到位——产品经理、设计师、技术领导都在吗？各自在发挥作用吗？
4. 检查授权方式——团队收到的是"问题"还是"功能清单"？
5. 给出改进建议，优先级排序
6. 输出团队优化方案

### 场景 C：用参考客户计划验证新产品

输入：用户在做一个全新产品，需要验证方向

执行步骤：
1. 帮用户选择 6 个参考客户——有真实需求、愿意投入时间、能代表目标市场
2. 设计合作方式——每周/双周和参考客户同步，展示进展、收集反馈
3. 定义"参考客户愿意为此付费"作为核心验证标准
4. 规划 3 个月的验证节奏
5. 设定退出标准——如果 6 个客户中不到 X 个愿意付费，重新评估方向
6. 输出参考客户验证计划

## 判断标准

- [ ] **问题先于方案** — 先明确了要解决的问题，再讨论方案
- [ ] **四种风险都考虑了** — 不只是"用户想不想要"，还有可用性、技术和商业
- [ ] **有产品发现** — 不是直接跳到"做"，而是先验证"值不值得做"
- [ ] **团队被赋能** — 团队收到的是问题，不是功能清单
- [ ] **有验证证据** — 决策基于用户测试/数据，不是HiPPO（最高薪人的意见）

## 常见误区

1. **把路线图当承诺**。产品路线图不应该是"Q3 做功能 A，Q4 做功能 B"——它应该是"Q3 解决问题 X，Q4 探索机会 Y"。功能可以变，问题是稳定的。
2. **产品经理 = 项目经理**。如果产品经理只是在写需求文档、排优先级、催进度——那他在做项目管理，不是产品管理。产品经理的核心工作是产品发现。
3. **只做用户说的**。用户会告诉你问题，但不会给你正确的解决方案。"亨利·福特说过，如果我问客户要什么，他们会说更快的马。"
4. **跳过发现直接开发**。"我们已经知道该做什么了"——这通常意味着最高薪的人拍了板。没有经过验证的"知道"，是假设伪装成事实。
5. **三个角色缺一个**。没有设计师，产品可用性差。没有技术领导参与发现，做出来的方案不可行。三角缺一不可。

## 组合使用

- **启示录 + 精益创业**：Cagan 的产品发现和 Ries 的构建-测量-学习高度互补。Cagan 提供"做什么、怎么验证"的系统方法，精益提供"快速循环、MVP"的执行理念。
- **启示录 + 设计冲刺**：Sprint 可以作为产品发现的一种高效方法——5天内完成从问题定义到原型测试。
- **启示录 + JTBD**：JTBD 帮你理解用户"为什么"买/用你的产品（Jobs），Cagan 的框架帮你把这个理解转化为产品决策。
- **启示录 + 妈妈测试**：Cagan 强调用户访谈的重要性，妈妈测试教你怎么正确地做用户访谈。

## Agent 指令

### 信息收集
在执行前需要知道：
- 用户的角色——产品经理、产品负责人、创始人？
- 产品阶段——0到1、增长期、成熟期？
- 当前团队配置——有设计师吗？技术领导参与产品决策吗？
- 当前最大的产品挑战——不知道做什么？做了没人用？团队效率低？
- 有没有数据支撑当前的决策？

先问角色和核心挑战。

### 输出格式

**产品发现计划**输出：
```
🔍 [产品/功能名] 产品发现计划

❓ 要解决的问题：[用一句话说清]

⚠️ 四种风险评估：
- 价值风险：🔴/🟡/🟢 — [说明]
- 可用性风险：🔴/🟡/🟢 — [说明]
- 可行性风险：🔴/🟡/🟢 — [说明]
- 商业可行性：🔴/🟡/🟢 — [说明]

🧪 验证计划（优先验证最高风险）：
1. [验证方法] — 解决 [哪个风险] — [时间]
2. [验证方法] — 解决 [哪个风险] — [时间]

✅ 开发信心标准：
[达到什么条件才有信心开始开发]
```

### 注意事项
- Cagan 的方法特别适合有一定规模的科技产品团队。纯个人项目或早期创业可能需要和精益创业组合
- 强调"问题先于方案"——很多产品经理来的时候已经有了方案，要引导他们退一步想问题
- 团队诊断要温和——如果用户的团队确实是"雇佣兵模式"，不要直接批评，而是引导看到另一种可能
